How to actually call Arbiter. It speaks the OpenAI chat-completions protocol, so
you keep your OpenAI client and change two things: the base URL, and your key.
This guide shows the common tasks with copy-paste examples. For the full endpoint
list see [api-reference.md](/docs/api-reference).

## 1. Get a key

Every request needs an Arbiter API key. Mint one from an email:

```bash
curl -X POST https://YOUR_HOST/v1/register \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com"}'
# -> { "api_key": "arb_..." }
```

(Or click through the app's `/start` flow, which shows the key.)

## 2. Point your client at Arbiter

The Arbiter key goes where your OpenAI key used to. The SDK sends it as
`Authorization: Bearer <key>`, which is exactly what Arbiter checks.

**Python (OpenAI SDK)**

```python
from openai import OpenAI

client = OpenAI(base_url="https://YOUR_HOST/v1", api_key="arb_...")

r = client.chat.completions.create(
    model="auto",                       # ignored; Arbiter picks the model
    messages=[{"role": "user", "content": "Calculate 88 * 12"}],
)
print(r.choices[0].message.content)
```

**Node (openai)**

```js
import OpenAI from "openai";
const client = new OpenAI({ baseURL: "https://YOUR_HOST/v1", apiKey: "arb_..." });
const r = await client.chat.completions.create({
  model: "auto",
  messages: [{ role: "user", content: "Calculate 88 * 12" }],
});
```

**curl**

```bash
curl https://YOUR_HOST/v1/chat/completions \
  -H "Authorization: Bearer arb_..." -H "Content-Type: application/json" \
  -d '{"model":"auto","messages":[{"role":"user","content":"hi"}]}'
```

## 3. Routing controls (budget, latency, cache)

Three optional, non-standard fields shape how a request is routed. They are all
stripped before the request reaches the runtime.

| Field | Effect |
|-------|--------|
| `arbiter_max_cost` (USD) | Route only among models whose estimated cost for this request is within the ceiling (falls back to the cheapest if none fit). |
| `arbiter_max_latency` (seconds) | Drop models whose learned mean latency exceeds the ceiling - for interactive paths. Untimed models pass. |
| `arbiter_no_cache` (bool) | Bypass the near-duplicate cache for this request (neither read nor write). |

Because they are non-standard, the OpenAI SDK sends them through **`extra_body`**:

```python
r = client.chat.completions.create(
    model="auto",
    messages=[{"role": "user", "content": "Summarize this in one line: ..."}],
    max_tokens=200,
    extra_body={
        "arbiter_max_cost": 0.0005,      # budget ceiling (USD)
        "arbiter_max_latency": 2.0,       # only models that answer under ~2s
        # "arbiter_no_cache": True,       # skip the near-duplicate cache
    },
)
```

Node:

```js
const r = await client.chat.completions.create({
  model: "auto",
  messages: [...],
  max_tokens: 200,
  // @ts-expect-error non-standard fields
  arbiter_max_cost: 0.0005,
  arbiter_max_latency: 2.0,
});
```

curl (they are just top-level fields in the body):

```bash
curl https://YOUR_HOST/v1/chat/completions \
  -H "Authorization: Bearer arb_..." -H "Content-Type: application/json" \
  -d '{"model":"auto","max_tokens":200,"arbiter_max_cost":0.0005,"arbiter_max_latency":2.0,
       "messages":[{"role":"user","content":"..."}]}'
```

## 4. Stream the response

Standard OpenAI streaming - set `stream: true`:

```python
stream = client.chat.completions.create(
    model="auto",
    messages=[{"role": "user", "content": "Write a short poem about the sea."}],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content or "", end="", flush=True)
```

Routing happens before the first token. After the stream, Arbiter emits one extra
`event: arbiter` with the final model, quality and cost (strict clients ignore
it). Read the serving model from that event: on streaming responses it is not a
header, because in-request failover can switch models after the headers are sent.

## 5. Read what Arbiter decided

Non-streaming responses include an `arbiter` object next to the usual fields:

```python
r = client.chat.completions.create(model="auto", messages=[...])
info = r.model_dump()["arbiter"]
# info["model"]         the model it routed to
# info["task"]          code / math / structured / factual / open
# info["difficulty"]    easy or hard (hard prompts route in their own bucket)
# info["mode"]          explore / exploit / escalate (cascade) / cache
# info["cost"]          real charge for the call, from the runtime's headers
# info["latency_ms"]    how long the call took
# info["cache"]         hit or miss  (+ cache_mode, cache_similarity on a hit)
# info["failover_from"] models that errored before this one served, or null
# info["cascaded_from"] cheap model escalated from, if the cascade fired, or null
# info["eligible_models"], info["budget_met"], ...
```

See [api-reference.md](/docs/api-reference#post-v1chatcompletions) for every field.

## 6. Send feedback (optional)

A thumbs up/down on an answer is the strongest quality signal the router has - it
overrides the model judge over a few votes. Target the `task` and `model` from the
response's `arbiter` block (use the difficulty sub-bucket, e.g. `code:hard`, when
`difficulty` was `hard`):

```bash
curl -X POST https://YOUR_HOST/v1/feedback \
  -H "Authorization: Bearer arb_..." -H "Content-Type: application/json" \
  -d '{"model":"deepseek-v4-flash","task":"code","rating":"up"}'
```

## 7. Manage your key

All authenticated with the key itself:

```bash
curl https://YOUR_HOST/v1/key -H "Authorization: Bearer arb_..."          # usage + status
curl -X POST https://YOUR_HOST/v1/key/pause  -H "Authorization: Bearer arb_..."
curl -X POST https://YOUR_HOST/v1/key/resume -H "Authorization: Bearer arb_..."
curl -X POST https://YOUR_HOST/v1/key/revoke -H "Authorization: Bearer arb_..."
```

## 8. Errors to handle

| Status | Meaning | What to do |
|--------|---------|------------|
| `401` | Missing or invalid key | Register or fix the `Authorization` header. |
| `403` | Key is paused | Resume it (`/v1/key/resume`). |
| `429` | Rate limit hit (50 / 6h or 600 / week) | Back off; a `Retry-After` header says how long. |
| `4xx` from upstream | The runtime/provider rejected the call | Surfaced with the upstream status; the failed call is not billed to your learning. |

## Notes

- The `model` you send is always ignored - choosing the model is the whole point.
- Everything here is the public API; the web Playground uses exactly these calls.
- Read-only endpoints (`/v1/report`, `/v1/overview`, ...) need no key.
